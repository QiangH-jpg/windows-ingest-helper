# 下一阶段计划：上传预处理与粗识别

**创建时间**: 2026-04-12  
**阶段**: 50 条原始素材规模前置层开发  
**优先级**: Windows 上传/预处理助手 > 服务端粗识别

---

## 一、问题背景

**用户场景**:
- 素材数量：约 50 条
- 总大小：约 5GB
- 素材类型：含人物专访、活动画面、服务动作等
- 当前痛点：
  - 手动上传 50 条素材耗时
  - 无法快速识别坏片
  - 无法快速分流专访类素材
  - 全量深审成本过高

---

## 二、当前链路的局限

| 问题 | 当前状态 | 50 条规模下影响 |
|------|---------|---------------|
| 上传方式 | 手动单条上传 | 耗时数小时 |
| 转码 | 服务端转码 | 带宽浪费 |
| 坏片检测 | 人工审看 | 50 条无法人工逐条看 |
| 专访分流 | 混入主池 | 稀释素材池质量 |
| AI 处理 | 全量深审 | 成本过高（¥15-25） |

---

## 三、下一阶段正式目标

建立分层处理架构：

```
Windows 上传/预处理 → 全量粗识别 → 候选收缩 → 精选深审 → 素材带优选
     ↓                      ↓           ↓          ↓          ↓
  720p proxy            类型粗分     过滤坏片    AI 深审     最终编辑
  坏片检测              专访分流     去重       评分       成片
```

**目标规模**: 50 条原始素材 / 约 5GB

---

## 四、分层链路设计

### 第 1 层：Windows 上传/预处理助手

**功能**:
1. 扫描用户指定素材目录
2. 识别原始文件元数据（文件名、大小、时长、分辨率、帧率、横竖屏）
3. 统一转码为 720p proxy
4. 生成原始文件 ↔ proxy 文件映射
5. 检测明显坏片（损坏/黑屏/极短/重复）
6. 分片上传到 TOS
7. 上传完成后生成素材清单 JSON
8. 保存完整元数据

**输出**:
- `manifest.json`（素材清单）
- `proxy/` 目录（720p proxy 文件）
- `uploads/` 目录（已上传记录）

### 第 2 层：服务端粗识别

**功能**:
1. 素材类型粗分（6 类）
   - 活动画面
   - 服务动作
   - 互动镜头
   - 合影/横幅
   - 专访/讲话
   - 空镜/弱价值镜头
2. 专访类素材分流
3. 明显重复合影素材初步聚类
4. 输出粗识别 JSON

**输出**: `coarse_recognition.json`

### 第 3 层：候选收缩

**功能**:
1. 过滤坏片
2. 过滤低价值镜头
3. 去重
4. 收缩到 20-30 条候选

**输出**: `candidate_list.json`

### 第 4 层：精选深审

**功能**:
1. AI 深度理解
2. 评分排序
3. 输出优选清单

**输入**: 候选清单（20-30 条）  
**输出**: `ranked_list.json`

### 第 5 层：素材带优选

**功能**:
1. 基于新闻稿选择镜头
2. 生成时间线
3. 输出成片
